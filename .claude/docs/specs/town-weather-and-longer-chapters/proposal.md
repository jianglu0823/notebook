# Proposal: 小镇天气系统 + 加长小说章节

## Why

智能体小镇当前的"天气"只是每日从季节词表里随机抽一个字符串(如"小雨"),既**没有连续性**(今天下雪、明天可能烈日,彼此无关联,也没有温度概念),也**不影响任何居民行为**——它仅作为日报叙事和居民 prompt 里的一句点缀。这让小镇缺乏"活着的环境"的沉浸感。同时,居民创作的小说章节被 prompt 写死为"80~140 字的精彩梗概",读起来像目录简介而非真正的小说,产物质量偏弱。本变更把天气升级为**有温度、会连续演变、并深度影响居民创作与社交的天气模型**,并让小说章节产出**800~1500 字的完整正文**,显著提升小镇的沉浸感与创作产物质量。

## What Changes

1. **连续天气模型**:引入带**温度(℃)**与**天气状况**的天气状态,天气**逐日演变**(今日天气受昨日天气影响 + 季节偏置 + 温度漂移),而非每日独立随机。天气状态持久化到世界设置与每日日报。
2. **天气影响结算**:天气深度联动每日结算——好天气(晴/秋高气爽)提升创作与社交概率;极端天气(台风/大雪/雷阵雨)降低外出型创作(画作/短片)与婚礼概率、但略微提升室内创作(小说/歌词),并可触发天气型突发事件(如台风致损、大雪封路)。
3. **天气注入叙事与居民 prompt**:日报主笔与居民自主行动的 prompt 中带上更丰富的天气信息(状况 + 温度 + 趋势)。
4. **前端天气可视化**:在小镇地图时钟区与日报详情中展示天气(emoji + 状况 + 温度),让天气"看得见"。
5. **加长小说章节**:居民 `novel` 技能产出的章节从"80~140 字梗概"改为**800~1500 字的完整章节正文**(标题 + 正文),并确保不被输出 token 上限截断。

## Scope

### In Scope
- `WorldSimEngine` 天气生成逻辑:从"季节随机抽词"升级为"逐日演变的天气状态(状况 + 温度)"
- 天气对每日结算(创作概率、婚礼概率、突发事件)的影响
- 天气信息注入日报叙事 prompt 与居民自主行动 prompt
- `WorldDailyReport` 与 `WorldSettings` 增加温度字段(schema 变更)
- 小说章节生成 prompt 加长至 800~1500 字完整正文
- 为 novel 章节创作显式设置 `max_tokens` 上限(约 3000,防止极端超长跑飞,并保证 800~1500 字不被截断)
- 前端小镇时钟区与日报详情的天气可视化(温度展示)
- `infra/mysql/init.sql` 对应建表列 + 手动迁移 SQL(提供给用户执行)

### Out of Scope
- 天气对经济/工资的直接影响(本次不改工资逻辑)
- 天气预报/未来多日预测的 UI
- 天气的地理分区(全镇统一天气,不做多地块差异)
- 歌词/画作/短片的长度调整(仅加长小说章节)
- 沙盒快进(`SandboxRunner`)内的天气模型(沙盒是独立纯规则引擎,不在本次范围)

## Impact

- **数据库**:`world_settings` 与 `world_daily_report` 各新增一个温度列(`temperature`),需手动迁移(init.sql 仅在全新库生效)。
- **每日结算行为**:创作/婚礼/突发事件的触发概率会因天气而波动——需保证在缺失温度(旧数据)时有安全回退。
- **Token/成本**:小说章节从 ~140 字增至 ~1500 字,单次 novel 创作的输出 token 上升约 5~10 倍;仍在智谱 GLM Flash 免费额度内,但每日结算总 token 会上升。
- **前端**:小镇时钟区与日报详情渲染需读取新增温度字段(旧日报无温度时优雅降级)。
- **向后兼容**:旧日报/旧设置无温度字段时,后端与前端均需容错(视为"未知温度",不报错)。

## Capabilities

- `town-weather`:小镇连续天气模型(温度 + 状况 + 逐日演变)、天气对结算的影响、天气可视化。
- `resident-novel-length`:居民小说章节产出加长为完整正文。
